Screen Share Tool

Live screen sharing for Unity WebGL and WebXR experiences, viewable right in the browser.

⚠️ Before you start

Open source under the MIT license and provided as-is, with no warranty. Back up your Unity project before importing, the way you would with any package.

What it is

A drop-in screen-sharing package for Unity WebGL and WebXR projects. Add it to a scene, point it at your own server, and visitors can share a desktop screen into the space and watch it together on a surface you place. It runs in the browser, so nobody downloads anything to view or share.

What works where

  • Watching a shared screen works on desktop, mobile, and WebXR VR browsers.
  • Sharing your screen works from desktop browsers.
  • It is browser-based, so it does not run in standalone (native) builds.

Download

Download the latest .unitypackage

Requires Unity 6 and the LiveKit WebGL SDK. The importer will offer to install the SDK for you.

Quick start in Unity

  1. Download the package above.
  2. In Unity, go to Assets > Import Package > Custom Package, choose the file, and import it.
  3. When the import finishes, a setup window offers to install the LiveKit WebGL SDK, which is the one dependency. Click Install LiveKit SDK. You can also add it yourself in Package Manager with Add package from git URL: https://github.com/livekit/client-sdk-unity-web.git#v2.0.0
  4. In your scene, use the menu Screen Share > Add Screen Share Setup. It creates the session, a screen surface, and the components that go with them.
  5. Set up your own token server using the guide below, then fill in the two addresses on the ScreenShareSettings asset.

Self-hosting the token server

The Screen Share Tool needs two things from you to connect: a LiveKit server URL and a small token endpoint that hands out signed access passes. This guide walks through running your own token endpoint on Vercel, so the keys and the cost are yours and nothing is shared with anyone else.

You don't need to be a backend developer to follow this. It's a single file, two settings, and a deploy button. The whole thing should take about fifteen minutes.

Does this work for my project?

This is a browser-based tool. It runs in experiences built with Unity 6.3+ LTS that play in a web browser, including browser-based VR (WebXR Export) and mobile browsers. It does not run in a standalone build, an app installed directly onto a headset like a Quest, or a native desktop or mobile app. Those don't have the browser layer this tool relies on.

Within the browser-based builds it does support, there's one split worth knowing up front:

  • Anyone can view a shared screen on a desktop browser, in a VR headset's browser (WebXR Export), or on a phone or tablet browser.
  • Sharing a screen works from desktop browsers only. A person in VR or on mobile can watch what's being shared, but the act of sharing is desktop-only.

So in a typical session someone on a desktop shares their screen, and everyone else, desktop, headset, or phone, watches.

(There's a short note at the end on what standalone requires, if that's what you're after.)

The request and response it expects

The Unity side of the tool sends one request and expects one answer. If you ever swap or rebuild your endpoint, this is what must be preserved, exactly.

The request it sends:

  • Method: POST
  • Header: Content-Type: application/json
  • Body:
{ "room": "some-room-name", "identity": "some-user-name", "canPublish": true }

canPublish is what decides whether that person can share their screen (true) or only watch (false).

The answer it expects back:

{ "token": "the-signed-access-pass" }

One thing that's easy to miss: this is a request sent from a web page on one address to your endpoint on a different address. Browsers protect those "cross-address" requests by first sending a quick permission check (an OPTIONS request) before the real one. Your endpoint has to answer that check first, not just the subsequent real request. The code below does this. If you don't know what any of it means, that's okay, you don't need to. Just don't delete the OPTIONS part (it's not optional).

Getting started

By the end of these steps you'll have:

  1. A LiveKit server, which gives you a server URL plus an API key and secret (free tier is fine to start).
  2. A free Vercel account, where the endpoint will live.
  3. Node.js installed on your computer, if you want to deploy from the command line. (You can also deploy straight from a Git repository without installing anything; both paths are below.)

Step 1 — Get your LiveKit server URL, key, and secret

The easy path: LiveKit Cloud.

  1. Sign in at https://cloud.livekit.io and create a new LiveKit project.
  2. In the left sidebar, open Settings → Project. Your address is the Project URL field, with a copy button beside it. One thing to watch for: LiveKit shows this URL without the wss:// in front, so you'll see something like your-project.livekit.cloud. The tool needs the full version, so add wss:// yourself to make it wss://your-project.livekit.cloud. Save that somewhere.
  3. In the left sidebar, open Settings → API keys and click Create key (top right). When the key is created, LiveKit shows you the key and its secret together, that one time only. Copy both right then and save them. If you already have a key but never saved its secret, you can't view the secret again, so just create a fresh key and copy that one instead.

Keep that secret private. It is like a password, and should never go anywhere in your Unity build, your web page, or any public repository. It only belongs in the server you're about to set up.

The self-hosting path. If you'd rather run your own LiveKit server (on your own machine or a host you control), follow LiveKit's self-hosting deployment docs. You'll end up with the same three things: a wss:// server URL pointing at your deployment, an API key, and an API secret. Once you have those, the rest of this guide is identical. The only difference is your server URL is your own address instead of a .livekit.cloud one.

Step 2 — Create the function files (on your computer)

Vercel is a free place to run small pieces of web code like this token endpoint, without setting up or maintaining an entire server yourself. If you don't already have an account, go to https://vercel.com and sign up. The free Hobby plan is plenty for this, and it doesn't ask for a credit card. Signing up with your GitHub account is the smoothest choice, since it makes the deploy step later a one-click import.

One thing to note: Vercel's free Hobby plan is meant for personal, non-commercial projects. If you're planning to run this as part of something commercial, check Vercel's current plan terms to see whether you need a paid tier.

The function's code doesn't get typed into the Vercel website. It lives in files on your own computer, and you'll send them up to Vercel when you deploy in Step 4. So on your computer, make a folder for your token server with this structure:

your-token-server/
  api/
    livekit-token.js
  package.json

your-token-server is any folder name you like; api and package.json must be named exactly as shown, and livekit-token.js can be renamed but its name becomes the end of your endpoint URL.

Put the following into package.json:

{
  "type": "module",
  "dependencies": {
    "livekit-server-sdk": "^2.15.4"
  }
}

Download package.json

Put the following into api/livekit-token.js:

// api/livekit-token.js
// Issues a signed LiveKit access token. POST room + identity in, signed token out.
// The API key and secret are read from environment variables and never leave the server.

import { AccessToken } from 'livekit-server-sdk';

// Browser origins allowed to call this endpoint. An "origin" is the scheme + host
// with no path, e.g. https://unreality3d.com. Add the address(es) your published
// experience is served from. Keep the localhost entries while testing.
const ALLOWED_ORIGINS = new Set([
  'https://unreality3d.com',
  'https://www.unreality3d.com',
  'http://localhost:3000',
  'http://localhost:8080'
]);

const TOKEN_TTL = '6h';

function applyCorsHeaders(req, res) {
  const origin = req.headers.origin;
  if (origin && ALLOWED_ORIGINS.has(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Vary', 'Origin');
    res.setHeader('Access-Control-Allow-Methods', 'POST, OPTIONS');
    res.setHeader('Access-Control-Allow-Headers', 'Content-Type');
    res.setHeader('Access-Control-Max-Age', '3600');
  }
}

function readBody(req) {
  if (!req.body) return {};
  if (typeof req.body === 'string') {
    try {
      return JSON.parse(req.body);
    } catch {
      return {};
    }
  }
  return req.body;
}

export default async function handler(req, res) {
  applyCorsHeaders(req, res);

  if (req.method === 'OPTIONS') {
    res.status(204).end();
    return;
  }

  if (req.method !== 'POST') {
    res.status(405).json({ error: 'Method not allowed' });
    return;
  }

  try {
    const apiKey = process.env.LIVEKIT_API_KEY;
    const apiSecret = process.env.LIVEKIT_API_SECRET;

    if (!apiKey || !apiSecret) {
      res.status(500).json({ error: 'Server is missing LiveKit credentials' });
      return;
    }

    const body = readBody(req);
    const room = typeof body.room === 'string' ? body.room.trim() : '';
    const identity = typeof body.identity === 'string' ? body.identity.trim() : '';
    const canPublish = body.canPublish === true;

    if (room.length === 0 || identity.length === 0) {
      res.status(400).json({ error: 'room and identity are required' });
      return;
    }

    // OPEN ENDPOINT (intentional for v1): anyone who can reach this can request a
    // token, including a screen-sharing token (canPublish). That's fine for a small,
    // trusted, private group. Before opening it to a wider or public audience, add a
    // caller check here (for example, verify a signed login your host issues) before
    // granting canPublish.
    const at = new AccessToken(apiKey, apiSecret, {
      identity,
      ttl: TOKEN_TTL
    });

    at.addGrant({
      roomJoin: true,
      room,
      canSubscribe: true,
      canPublish: canPublish,
      canPublishData: canPublish
    });

    const token = await at.toJwt();
    res.status(200).json({ token });
  } catch (error) {
    console.error('livekit-token error:', error && (error.code || error.message));
    res.status(500).json({ error: 'Token generation failed' });
  }
}

Download livekit-token.js

This checks if the request is a real POST, reads room, identity, and canPublish, signs a six-hour access pass with your LiveKit key and secret, and sends back { "token": "..." }. The publishing permission is tied directly to canPublish, so a watcher can never accidentally be handed a sharing pass.

Step 3 — Set your allowed origins

Open api/livekit-token.js and look at the ALLOWED_ORIGINS list at the top. This is the guest list. Only web pages served from an address on this list are allowed to ask your endpoint for a token. Everything else gets turned away.

Add the origin of the URL where your published experience runs. A couple of examples:

  • If you serve from your own site, add that site's https:// address.
  • If you serve from GitHub Pages, add your GitHub Pages address, which is https://yourusername.github.io (just the scheme and host; the repository name that comes after it in the path is not part of the origin, so leave it off).
  • Keep the localhost entries while you're testing locally, and remove them later if you like.

An origin is the scheme and host only. No trailing slash, no path. https://unreality3d.com is right; https://unreality3d.com/ or https://unreality3d.com/yourname is not.

Do this before you deploy, since these origins are baked into the file you upload. If you change them later, edit the file and deploy again.

Step 4 — Deploy and get your endpoint URL

This is the step that actually creates your project on Vercel and puts your code online. Pick whichever path you're comfortable with.

From the command line: open a terminal in your token-server folder and run npx vercel. The first run asks a few quick questions: log in when prompted, say yes to set up and deploy, accept your account as the scope, let it create a new project (press Enter to accept the suggested name), and when it asks which directory your code is in, press Enter for the current folder. That gives you a preview. Run npx vercel --prod to publish the live version.

From a Git repository: push the folder to GitHub, then in Vercel choose "Add New Project," import the repo, and deploy. The import creates the project for you.

When it's live, your endpoint URL is your project's address followed by the function path:

https://your-project-name.vercel.app/api/livekit-token

That full URL is what Unity needs. One heads-up: the endpoint won't hand out tokens yet, it still needs your LiveKit credentials, which is the next step.

Step 5 — Add your LiveKit credentials, then redeploy

Your key and secret don't go in any code. They go into your Vercel project's settings, where they stay secure on the server. Now that your project exists from Step 4, you can add them:

  1. In your Vercel project, open Settings → Environment Variables.
  2. Add LIVEKIT_API_KEY with your LiveKit key as the value.
  3. Add LIVEKIT_API_SECRET with your LiveKit secret as the value.
  4. Deploy once more so the new values take effect: run npx vercel --prod again, or if you used the Git path, use Vercel's Redeploy button on the latest deployment.

That's why the code reads process.env.LIVEKIT_API_KEY and process.env.LIVEKIT_API_SECRET instead of having the actual values written in. The names have to match exactly. After this redeploy, your endpoint is live and issuing tokens.

Step 6 — Paste both values into the asset in Unity

Open your ScreenShareSettings asset in the Unity Editor. (If you don't have one yet, create it from the right-click menu in the Project window: Assets > Create > Screen Share > Settings, then assign it to your ScreenShareRoom.) Fill in the two fields:

  • Server Url: your wss://... address from Step 1.
  • Token Endpoint Url: the https://your-project-name.vercel.app/api/livekit-token address from Step 5.

Nothing else on the Unity side changes. The tool already knows how to call the endpoint in the shape described up top. Save the asset, build, and you'll be connecting to your own token server.

Checking that it works

Before building in Unity, you can test the endpoint from your computer's terminal. If you're testing right after your first deploy, the most likely thing still undone is Step 5, adding your LiveKit credentials and redeploying, so start there if the token test below comes back with a credentials error.

Check the permission step (the preflight):

curl -i -X OPTIONS https://your-project-name.vercel.app/api/livekit-token \
  -H "Origin: https://unreality3d.com" \
  -H "Access-Control-Request-Method: POST"

You want a 204 response, and an Access-Control-Allow-Origin header that echoes back your origin. This check passes whether or not your credentials are in place, since it's only testing the allow list. If that header is missing, the origin isn't on your allow list; check Step 3 for a typo or a stray slash.

Get a real token:

curl -i -X POST https://your-project-name.vercel.app/api/livekit-token \
  -H "Content-Type: application/json" \
  -H "Origin: https://unreality3d.com" \
  -d '{"room":"test-room","identity":"test-user","canPublish":true}'

You want a 200 response with a long token string. If you get a 500 saying credentials are missing, that's Step 5; add the two environment variables and redeploy, then try again. If you get a 400, the body is missing room or identity.

A note on security

This endpoint is open on purpose for a test version. Anyone who knows the URL and is calling from an allowed origin can request a token, including a screen-sharing one. For a small private group of people you trust, that's a reasonable place to start, and it keeps the setup simple.

If you later open this up to a wider or public audience, that's the time to add a caller check. The spot for it is marked in the code above, right before the token is granted: confirm the person is who they say they are (for example, by checking a signed login your host already issues) before handing out a sharing pass. Adding that check doesn't change anything on the Unity side, since the request and answer stay the same shape.

Going further: standalone VR

If you're building a standalone app, one installed directly onto a headset rather than opened in its browser, the Screen Share Tool itself won't run there, because it relies on the browser connection that standalone builds don't have.

The good news for anyone interested in learning that path: the token server you just built is reusable as-is. A standalone build still needs a token, still asks for it the same way, and still connects to the same LiveKit server URL. What changes is only the Unity side.

LiveKit's Unity connection comes in two separate parts, one made for browser builds (which works with De-Panther's WebXR Export enabled VR) and one made for standalone builds. Each only runs on the kind of build it was made for. This tool uses the browser kind. A standalone build needs the standalone part instead (or in addition to the browser part, if you want to support both), and since the standalone part doesn't run in a browser, adding it covers the standalone side but not the browser side. Covering both audiences requires setting up and maintaining both.

So if standalone VR is your goal, treat this guide as the back half already done. The front half, wiring up the standalone client in Unity, is its own project, and it's outside what this tool and doc cover.
